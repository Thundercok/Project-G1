# Asset pipeline

Every 3D asset in this project — both characters, their rigs, their animations, and
the crowbar — is generated by Python scripts in [`Tools/blender/`](../Tools/blender).
There are no hand-sculpted meshes. This means any asset can be regenerated,
re-proportioned, or re-colored by editing numbers in a script and re-running it.

## The scripts

| Script | Produces |
|---|---|
| `build_character.py` | A character as beveled primitives + studio + turnaround renders + static FBX |
| `rig_character.py` | Armature, skinning, Idle/Walk actions, animated FBX for Unity |
| `build_crowbar.py` | The crowbar mesh, a beauty render, static FBX |

All run headless. Typical regeneration flow (Blender 5.x, macOS path shown):

```bash
B=/Applications/Blender.app/Contents/MacOS/Blender
G=~/halflife-like-game            # working dir containing blender/, renders/
U=<repo>/Assets/G1/Models

# 1. build the character (writes protagonist.blend + turnaround renders)
$B --background --python Tools/blender/build_character.py -- protagonist $G

# 2. rig + animate + export FBX into Unity
$B --background $G/blender/protagonist.blend \
   --python Tools/blender/rig_character.py -- protagonist $G $U

# crowbar (build + render + export in one step)
$B --background --python Tools/blender/build_crowbar.py -- $G $U
```

Same commands with `villain` for the second character. After export, Unity picks the
FBX up automatically; re-run **G1 → Build Test Scene** to refresh importer settings
(clip names, looping) if the animation set changed.

## Design rules the scripts follow

- **Units are meters, characters ~1.8 m tall**, feet at Z = 0, facing −Y in Blender
  (this exports to +Z forward in Unity with the settings used).
- **Retro look**: beveled boxes/cylinders/spheres, flat shading, ~3–6k triangles per
  character. Solid-color Principled materials only — no textures, so FBX materials
  import into Unity losslessly.
- **Pivot placement matters**: the crowbar's origin is at the grip so the viewmodel
  can be mounted without offsets.

## FBX export settings that Unity likes

Used by all scripts (see `export_scene.fbx` calls):

- `apply_scale_options="FBX_SCALE_ALL"`, `apply_unit_scale=True` — imports at scale 1
- `axis_forward="-Z"`, `axis_up="Y"`
- `add_leaf_bones=False`, `use_armature_deform_only=True` — clean bone list
- `bake_anim_use_nla_strips=True` — each NLA track becomes an FBX take
  (→ a Unity AnimationClip); takes are named `<Rig>|<Action>` and renamed to
  `Idle` / `Walk` by the importer configuration in `G1SceneBuilder`.
- `mesh_smooth_type="FACE"` — avoids Unity smoothing-group warnings

## Hard-won gotchas (read before editing the scripts)

1. **`bpy.ops.object.transform_apply(scale=True)` also applies location and rotation**
   (their defaults are `True`). Always pass
   `transform_apply(location=False, rotation=False, scale=True)`. The original bug
   baked every object's origin to the world origin, so any later object-space
   rotation orbited the world origin — parts ended up under the floor.
2. **Emission clips to white.** With the Standard view transform, emission strength
   above ~1.5 washes out to pure white and vanishes against light surfaces. The
   villain's glowing eyes use strength 0.55; the Geiger screen 1.2 against a dark
   housing.
3. **`Action.fcurves` is gone in Blender 5.x** (slotted actions). Use
   `action.layers[0].strips[0].channelbag(action.slots[0]).fcurves`.
4. **`to_track_quat` rolls unpredictably** for near-vertical directions. Don't
   inherit its roll for detail parts that must face a specific direction — place
   them axis-aligned instead (see the Geiger screen).
5. **Overexposure desaturates.** If hazard orange renders as yellow, lower the light
   energy before touching the material color.
